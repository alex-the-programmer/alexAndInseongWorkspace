# Backend Guidelines

## Architecture
All backends use a two-layer architecture:

- **API layer** — GraphQL resolvers (Apollo Server on Express). Resolvers map operations to interactions; no business logic or direct DB access in resolvers.
- **Service layer** — `interactions/` folder. Interactions are single-responsibility functions grouped by primary resource. They access data through Prisma ORM.
  - Keep interactions small and composable. Prefer several focused interactions calling each other over one large one.
  - When writing a new Prisma query in an interaction, verify it is covered by an existing index in `schema.prisma`. Add indexes if needed (see Database section).

## Database (Prisma)
- Prisma (`schema.prisma`) is used in all backend repos. `commerce-platform-scrapers` introspects rather than owning migrations — see that repo's CLAUDE.md.
- All database changes must be explicitly mentioned in implementation plans with exact table and column names before being applied.
- Make schema changes in `schema.prisma` first, then run: `npx prisma migrate dev --name "<meaningful-name>"`.
- For index types Prisma cannot express (e.g. GIN), generate an empty migration with `--create-only`, add raw SQL, mark the index as externally managed in the schema, then apply.
- Never reset the database. If Prisma reports a mismatch that requires a reset, stop and delegate to a human — unless you introduced the mismatch yourself in the current session.
- Do not create concurrent indexes — Prisma migrations do not support them and the migration will fail.

## GraphQL
- After any change to `schema.graphql`, run `npm run codegen` to regenerate types.
- Never use the `ID` scalar type; use `EncodedID` instead.

## Pre-push checks
Run `npm run lint` (`tsc --noEmit`) and `npm run build` before pushing. Fix all type and build errors locally — CI/Vercel run the same `tsc` step and will fail on the same issues.

## Testing
Run tests with `npm test`. Coverage report with `npm run test:coverage`.

Two distinct test layers — never mix them:

**Interaction tests** (`src/__tests__/interactions/`)
- Test real DB behavior using `jest-prisma` auto-rollback (each test runs inside a rolled-back transaction).
- Seed data with factories from `src/__tests__/factories/index.ts` (generated by `@quramy/prisma-fabbrica`).
- Assert on the interaction's return value and DB side-effects.
- No mocking of Prisma.
- Mocking other interaction modules is allowed when isolating the interaction under test (e.g. mock a downstream LLM callee). The test must still exercise this interaction's own Prisma reads/writes directly.

**GraphQL resolver tests** (`src/__tests__/graphql/`)
- Test resolver logic only: argument decoding, context extraction, interaction dispatch, and response field mapping.
- Mock the interaction modules with `jest.unstable_mockModule` before any dynamic import that transitively loads them. Then dynamically import the test server helper.
- Assert that the interaction mock was called with the correct decoded arguments (e.g. `EncodedID` → number, `auth.userId` → clerkUserId) and that the resolver maps the return value to the correct GraphQL fields.
- No DB access, no factories, no `jestPrisma` in GraphQL tests.

Example GraphQL test pattern:
```ts
jest.unstable_mockModule("../../interactions/cart/getMyCart.js", () => ({
  default: jest.fn(),
}));
const { buildTestServer, makeContext } = await import("./helpers/buildTestServer.js");
const mockGetMyCart = jest.mocked((await import("../../interactions/cart/getMyCart.js")).default);
```
